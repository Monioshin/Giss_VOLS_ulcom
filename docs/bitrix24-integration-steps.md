# Пошаговая интеграция Bitrix24 с GIS

Краткий чеклист: что сделать **в каком порядке**, чтобы связка заработала. Технические детали API — в [bitrix-integration.md](./bitrix-integration.md).

---

## Этап 0. Предусловия

1. **GIS запущен:** backend (порт 4000) и frontend (5173 или ваш домен).
2. **Bitrix и GIS «видят» друг друга по сети:**
   - роботы Bitrix вызывают **ваш** URL GIS (HTTPS в production);
   - сервер GIS вызывает **ваш** портал Bitrix (`*.bitrix24.ru/rest/...`).
3. Есть **администратор GIS** (логин в веб-интерфейс) — для выпуска embed-токена и настройки баз.

---

## Этап 1. Входящий webhook в Bitrix (GIS → Bitrix REST)

Это URL, по которому **ваш сервер GIS** дергает REST Bitrix (`crm.deal.update`, `tasks.task.add`).

1. В Bitrix24: **Приложения** → **Вебхуки** → **Входящий вебхук** (или **Разработчикам** → **Другое** → входящий webhook — зависит от интерфейса портала).
2. Создайте webhook с правами минимум:
   - **CRM (сделки)** — изменение сделок;
   - **Задачи** — создание задач (для сценария «авария»).
3. Скопируйте URL вида  
   `https://ВАШ_ПОРТАЛ.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/`  
   **Важно:** в конце должен быть слэш `/`.

4. На сервере GIS в переменных окружения backend укажите:

```bash
BITRIX_REST_WEBHOOK_URL=https://ВАШ_ПОРТАЛ.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/
```

5. Перезапустите backend GIS.

**Проверка:** пока нет секрета из этапа 2, входящие вызовы из Bitrix в GIS не настраивайте — сначала доведите env до конца этапов 2–3.

---

## Этап 2. Секрет для вызовов Bitrix → GIS

1. Придумайте длинную случайную строку (например 32+ символа) — это общий секрет между роботом Bitrix и GIS.

2. На сервере GIS:

```bash
BITRIX_INBOUND_SECRET=вставьте_сюду_секрет
```

3. Перезапустите backend.

**Правило:** любой запрос из Bitrix к `https://ваш-gis.../integrations/bitrix/...` должен передавать этот секрет:
- заголовок `X-GIS-Bitrix-Secret: <секрет>`, **или**
- query-параметр `?secret=<секрет>` (хуже с точки зрения логов, но проще в некоторых роботах).

---

## Этап 3. Публичный URL фронтенда GIS

Нужен для ссылок в задачах Bitrix («открыть карту») и для поля `embedUrl`.

```bash
GIS_PUBLIC_URL=https://gis.ваш-домен.ru
```

Для локальной отладки часто достаточно `http://localhost:5173` (по умолчанию в коде).

Перезапустите backend после изменения.

---

## Этап 4. Пользовательские поля (UF) в карточке сделки

В **CRM → Настройки → Пользовательские поля → Сделка** создайте поля с **символьными кодами** (можно свои — тогда пропишите их в env, см. [bitrix-integration.md](./bitrix-integration.md)).

Рекомендуемые коды «из коробки» (совпадают с дефолтами в GIS):

| Символьный код (поле сделки) | Тип (рекомендация) | Зачем |
|------------------------------|--------------------|--------|
| `UF_CRM_GIS_PROJECT_ID` | Число | ID проекта GIS после «сделка выиграна» |
| `UF_CRM_GIS_FIBER_ORDER_ID` | Число | ID заказа по волокну из GIS |
| `UF_CRM_ROUTE_LENGTH_M` | Число / строка | Длина маршрута, м |
| `UF_CRM_ROUTE_FREE_FIBERS` | Число | Мин. свободных волокон на маршруте |
| `UF_CRM_ROUTE_FOUND` | Строка / список `Y`/`N` | Найден ли маршрут |

Если коды другие — задайте переменные `BITRIX_UF_*` на backend (см. справочник в [bitrix-integration.md](./bitrix-integration.md)).

---

## Этап 5. Ответственный за задачи «Авария» (опционально, но желательно)

В Bitrix откройте профиль пользователя-инженера и запомните **числовой ID** (часто виден в URL или в REST).

```bash
BITRIX_DEFAULT_TASK_RESPONSIBLE_ID=123
```

Если не задать — по умолчанию используется `1` (первый пользователь портала).

Перезапустите backend.

---

## Этап 6. Проверка, что GIS «видит» конфиг

С машины, где доступен ваш API (или с сервера):

```bash
curl -sS "https://ВАШ-GIS-API/integrations/bitrix/field-catalog" \
  -H "X-GIS-Bitrix-Secret: ВАШ_BITRIX_INBOUND_SECRET"
```

Ожидается JSON со списком полей и переменных.  
Если `503` и текст про отсутствие секрета — не задан `BITRIX_INBOUND_SECRET`.  
Если `403` — неверный секрет в заголовке.

---

## Этап 7. Сценарий «Проверка маршрута» (робот Bitrix → GIS)

1. В GIS на карте найдите **id двух узлов** (муфта/кросс) — начало и конец проверки. Это целые числа из базы.
2. В Bitrix создайте **исходящий вебхук** или **робот сделки** с действием «HTTP запрос к другому сайту» (название может отличаться):
   - **URL:** `https://ВАШ-GIS-API/integrations/bitrix/route-check`
   - **Метод:** POST  
   - **Тело (JSON),** подставьте свои id и id сделки:

```json
{
  "start_node_id": 10,
  "end_node_id": 20,
  "required_free_fibers": 1,
  "deal_id": 99,
  "update_bitrix_deal": true
}
```

   - **Заголовок:** `X-GIS-Bitrix-Secret: <ваш BITRIX_INBOUND_SECRET>`  
   - `Content-Type: application/json`

3. Убедитесь, что в Bitrix задан **входящий webhook** (этап 1), иначе GIS не сможет вызвать `crm.deal.update` — в ответе GIS будет блок `bitrix` с ошибкой.

4. Проверьте в карточке сделки, что UF-поля заполнились.

**Подсказка:** узлы «рядом с адресом» можно получить через `POST .../integrations/bitrix/nearest-endpoints` с `lat`/`lng`, если координаты уже есть в CRM или из геокодера.

---

## Этап 8. Карта во фрейме в карточке сделки (embed)

1. Войдите в GIS как **администратор**.
2. Выпустите токен (пример через curl; подставьте JWT после обычного входа в GIS):

```bash
curl -sS "https://ВАШ-GIS-API/integrations/bitrix/embed-token?highlight_edge_id=12" \
  -H "Authorization: Bearer ВАШ_JWT_АДМИНА"
```

3. В ответе возьмите **`embedUrl`** — это ссылка на frontend с `#embed=...`.
4. В Bitrix: локальное приложение / **встраивание** / **iframe** на вкладке сделки — вставьте этот URL (или свой плейсмент `CRM_DEAL_DETAIL_TAB`, если делаете приложение маркетплейса).

Токен живёт **около 15 минут**; для постоянной вкладки нужен либо периодический перевыпуск токена роботом, либо доработка под SSO (вне текущего MVP).

---

## Этап 9. Авария GIS → задача Bitrix и обратно

**GIS → Bitrix (уже в коде):** при смене статуса ВОЛС на `ACCIDENT` создаётся задача, если задан `BITRIX_REST_WEBHOOK_URL`.

**Bitrix → GIS (настройте робот):** при закрытии задачи вызовите:

`POST https://ВАШ-GIS-API/integrations/bitrix/webhook`  
с секретом и телом, например:

```json
{
  "type": "task_completed",
  "workspace_slug": "имя_активной_базы_gis",
  "edge_id": 12,
  "new_cable_status": "READY"
}
```

**Критично:** `workspace_slug` должен совпадать с **активной** рабочей базой в GIS (как в списке баз в интерфейсе). Если база другая — переключите активную в GIS или передавайте правильный slug.

## Этап 10. Дополнительные сценарии (по необходимости)

| Задача | Действие |
|--------|----------|
| Сделка выиграна → проект в GIS | Робот → `POST /integrations/bitrix/deal-won` (см. пример в [bitrix-integration.md](./bitrix-integration.md)) |
| Стройка: смена статуса всех ВОЛС проекта | Робот → `POST /integrations/bitrix/project-edges-status` |
| Заказ по волокну + обновление сделки | В GIS при создании заказа укажите `bitrix_deal_id` (поле в форме заказа или в JSON API) |

## Частые ошибки

| Симптом | Что проверить |
|---------|----------------|
| `403 Forbidden` на `/integrations/bitrix/*` | Заголовок `X-GIS-Bitrix-Secret` и значение `BITRIX_INBOUND_SECRET` |
| `503` про секрет | Не задан `BITRIX_INBOUND_SECRET` |
| Bitrix не обновляет сделку | `BITRIX_REST_WEBHOOK_URL`, права webhook, корректные коды UF |
| `409` про workspace | В теле webhook указан другой `workspace_slug`, чем активная база в GIS |
| Ошибка входа в GIS в браузере | Backend не запущен или упал при старте — смотрите логи `node` |

## Минимальный чеклист перед демо

- [ ] `BITRIX_INBOUND_SECRET` задан, backend перезапущен  
- [ ] `BITRIX_REST_WEBHOOK_URL` со слэшем в конце, backend перезапущен  
- [ ] `GIS_PUBLIC_URL` указывает на реальный адрес фронта  
- [ ] UF-поля в сделке созданы с нужными символьными кодами  
- [ ] Один раз проверен `field-catalog` с секретом  
- [ ] Робот `route-check` отрабатывает на тестовой сделке  
- [ ] (Опционально) embed-токен открывается в iframe  
- [ ] (Опционально) закрытие задачи дергает `/integrations/bitrix/webhook` с верным `workspace_slug`  

После этого интеграция считается **настроенной для показа** по основным сценариям плана.
