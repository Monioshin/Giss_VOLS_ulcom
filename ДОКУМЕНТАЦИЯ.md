# GIS · ВОЛС — документация программы

Локальная веб-система для учёта и проектирования линейно-кабельной инфраструктуры (ЛКС) и волоконно-оптических линий связи (ВОЛС): карта, паспорта объектов, схемы сварки в муфтах и кроссах, маршруты, аналитика и администрирование.

**Стек:** React 19 + TypeScript + Vite (frontend), Express 5 + SQLite (backend), опционально Electron (desktop).

## 1. Запуск и окружение

**Установка: сервер (backend) + приложение Windows (.exe):** [УСТАНОВКА.md](УСТАНОВКА.md).

### Требования

- Node.js 18+ (рекомендуется LTS)
- npm

### Backend

```bash
cd backend
npm install
npm run dev    # nodemon, порт 4000 по умолчанию
# или
npm start
```

База данных: `backend/data/gis.sqlite` (создаётся автоматически, режим WAL).

Переменные окружения:

| Переменная | Назначение |
|------------|------------|
| `PORT` | Порт API (по умолчанию `4000`) |
| `JWT_SECRET` | Секрет JWT (в production обязательно сменить) |
| `BITRIX_INBOUND_SECRET` | Секрет для входящих вебхуков Bitrix → GIS (см. [docs/bitrix-integration.md](docs/bitrix-integration.md)) |
| `BITRIX_REST_WEBHOOK_URL` | URL REST-вебхука Bitrix для исходящих вызовов из GIS (`crm.deal.update`, `tasks.task.add`) |
| `GIS_PUBLIC_URL` | Публичный URL фронтенда для ссылок в задачах и `embedUrl` |

### Frontend (браузер)

```bash
cd frontend
npm install
npm run dev    # Vite, http://127.0.0.1:5173
```

API по умолчанию: `http://localhost:4000` (константа в `frontend/src/App.tsx`).

### Desktop (Electron) — основной режим для офиса

На сервере только backend; на ПК Windows — приложение из `npm run build:desktop` (см. [УСТАНОВКА.md](УСТАНОВКА.md)).

```bash
cd frontend
npm run dev:desktop   # разработка: Vite + Electron + локальный backend
```

Адрес API в приложении: файл `%APPDATA%/GIS VOLS/gis-desktop.json` (`apiUrl`) или переменная `VITE_API_URL` при сборке.

### Сборка production

```bash
cd frontend && npm run build
cd backend && npm start
```

## 3. Модель данных

### Проекты (`projects`)

Группировка участков ВОЛС. У каждого оптического участка обязателен `project_id`.

### Узлы (`nodes`)

| Тип | Код | Назначение |
|-----|-----|------------|
| Телефонная канализация (точка) | `TK` | Опора канализации, привязка муфты |
| Муфта | `MUFTA` | Сварка ВОЛС, схема в `passport_data.splice_v1` |
| Пикет | `PIKET` | Метка трассы |
| Кросс | `KROSS` | Распределительный кросс, порты и сварки |

Поля: `name`, `lat`, `lng`, `parent_tk_id` (для `MUFTA` — id ТК; **на одном ТК может быть несколько муфт** — сервер выстраивает их по кругу вокруг колодца; проверка ≤ 2 м от центра ТК), `passport_data` (JSON).

### Участки (`edges`)

| Тип | Код | Концы | Проект |
|-----|-----|-------|--------|
| Канализация | `KANALIZACIYA` | ТК ↔ ТК | не требуется (`null`) |
| ВОЛС | `OPTOVOLOKNO` | муфта/кросс ↔ муфта/кросс | обязателен |

Общие поля: `geometry` — массив `[lat, lng]`, `length_m`, `passport_data`.

Для `OPTOVOLOKNO`: `cable_name`, `total_fibers`, `used_fibers`, `cable_status`:

| Статус | Смысл | Цвет на карте |
|--------|--------|----------------|
| `READY` | Готов | оранжевый |
| `IN_WORK` | В работе | жёлтый |
| `CONSTRUCTION` | Строится | фиолетовый |
| `OFFLINE` | Не работает | серый |
| `ACCIDENT` | Авария | розовый, усиленное свечение |

В `passport_data` оптического участка может храниться детальная занятость волокон (`fiber_usage`) и схема сварки на стороне муфт (`splice_v1` в узле).

### Заказы по волокну (`fiber_orders`)

Запись маршрута между двумя муфтами с увеличением `used_fibers` на проходящих участках.

### Пользователи (`users`)

Роли: `ADMIN`, `USER`. JWT в заголовке `Authorization: Bearer <token>`.

## 5. Карта

### Инструменты (панель слева от карты)

| Инструмент | Действие |
|------------|----------|
| Выбор | Клик по объекту, перетаскивание узла |
| ТК / Муфта / Кросс / Пикет | Создание узла по клику |
| Канализация / ВОЛС | Рисование линии по точкам (двойной клик или завершение) |
| **Канализация по ТК** | Участок между двумя колодцами по **названиям** начального и конечного ТК и заданной **длине** (м). На карте — прямая между координатами ТК; в поле `length_m` и паспорте (`catalog_length_m`) — введённая длина. API: `POST /edges/kanal/link`. **Пакетно из Excel:** в том же окне — «Скачать шаблон» / «Загрузить Excel»; колонки **точка А**, **точка Б**, **длина** (названия ТК как в базе), опционально тип трубы / диаметр / примечание; API `POST /edges/kanal/link/bulk`. Ручное рисование с изгибами — инструмент «Канализация». |
| **Линейка** | Измерение расстояния по цепочке кликов; подписи сегментов и сумма; ПКМ / Esc — сброс; **в базу не сохраняется** |

**Привязка:** узлы ищут ближайшие цели в радиусе ~30 м (`SNAP_METERS`). Муфта привязывается к ТК (`parent_tk_id`); при добавлении нескольких муфт на один ТК они автоматически раскладываются по окружности вокруг колодца (не в одной точке). ВОЛС — к муфте/кроссу.

### Слои и подложка

- Включение/выключение канализации и ВОЛС.
- Подложка: OSM (улицы), спутник Esri, гибрид (спутник + подписи).
- **Кластеризация отключена:** сдалека видны линии (канализация / ВОЛС), при приближении — отдельные узлы на canvas.
- Подписи: при наведении; при zoom ≥ 16 — лимит из настроек (с ужатием при тысячах точек в кадре).
- При **>2000 узлов** в базе включается режим области: данные подгружаются по bbox (`GET /map/viewport?bbox=…&zoom=…&types=…`). Сеть — после окончания pan/zoom с debounce/throttle (на **удалённом API** — дольше, ~450/700 ms). По LAN первая порция области: `includeEdges=split` (узлы + канализация + ВОЛС в **одном** запросе), затем догрузка страниц. До ответа — кэш на экране. Фоновая синхронизация (`sync/summary`) обновляет списки и **лёгко** перезагружает область (`refreshMapLight`), без полной очистки кэша. Backend отдаёт JSON с **gzip** (порог 1 KB). Сохранение вида карты — в localStorage.
- На **zoom &lt; 16** API и клиент запрашивают только **ТК** (меньший payload); муфты и кроссы рисуются при zoom ≥ 16.
- В RAM держится ограниченный объём узлов/линий (см. `mapZoomConstants.ts`); при превышении — флаг «показана часть» в уведомлениях.
- **Участки ВОЛС и канализации** видны на всех масштабах; узлы детализации — с zoom ≥ 16. Канализация рисуется, если линия **пересекает** экран (в т.ч. прямая ТК–ТК с колодцами за кадром); при отдалении линии **не упрощаются** и подгружаются отдельным запросом (до 6×2000 на тип). При смене ступени zoom данные **перезапрашиваются** (кэш с LOD). Пустые `bbox_*`: `POST /database/repair-edge-bbox` (админ).
- Canvas узлов не перехватывает drag карты (`pointer-events: none`, клики через hit-test на карте).
- Легенда статусов ВОЛС.

### Статус карты в шапке

На вкладке **Карта** в верхней панели — иконка **уведомлений** (`MapStatusBell`): режим области, загрузка, усечение данных, сводка «в базе / в памяти». Тонкая полоска под шапкой — индикатор загрузки области (дождитесь её исчезновения после зума). На вкладке **База данных** при первой загрузке категории счётчик «из —» и скелетон строк вместо пустой таблицы. Подсказки **не** рисуются поверх карты.

### Маршруты

Панель маршрутов (если включена в настройках): поиск пути между двумя муфтами/кроссами с учётом свободных волокон (`GET /routes`). Найденные участки подсвечиваются зелёным.

### Трассировка волокна

Логический маршрут по сваркам и кроссу (`fiberRouteTrace.ts`) — подсветка цепочки на карте и на схеме сварки.

### Изгибы линии

Для выбранного участка — режим добавления промежуточных точек геометрии («изгиб по линии» из паспорта).

### Контекстное меню

ПКМ по объекту: удаление, перемещение, фильтр проекта (для проекта на карте). В каталоге БД — паспорт, карта, копирование id.

### Паспорт на карте

Drawer справа при выборе объекта. Сохранение — PATCH узла/участка/проекта через API.

## 6. Паспорт объекта

Файлы: `frontend/src/passport/PassportDrawer.tsx`, `schemas.ts`, `PassportRelations.tsx`, `EdgeOpticalFiberGrid.tsx`.

Вкладки паспорта:

- **Основное** — поля по типу объекта (ТК, муфта, кросс, пикет, кабель, канализация, проект).
- **Связи** — соседние узлы и участки, переход к другому паспорту.
- **Дополнительно** — заметки, JSON `passport_data` для опытных пользователей.

Для **ВОЛС** — сетка занятости волокон, статус участка, кнопка изгибов. Для **муфты/кросса** — переход в раздел «Сварка».

Данные паспорта — произвольный JSON в `passport_data`; типизированные читатели/писатели в `schemas.ts` (`readTkPassport`, `writeOpticalPassport`, и т.д.).

## 7. Сварка (муфта / кросс)

### Вход

Вкладка **Сварка** → выбор муфты или кросса. Либо кнопка в паспорте узла.

### Схема (`frontend/src/splice/`)

| Модуль | Роль |
|--------|------|
| `layout.ts` | Размещение волокон вокруг корпуса муфты по сторонам |
| `spliceDiagram.tsx` | SVG: точки волокон, связи, подписи |
| `linkGeometry.ts` | Кривые связей, waypoints, смещение пучка |
| `spliceSidePanel.tsx` | Список волокон, фильтры, пакетная сварка |
| `spliceToolbar.tsx` | Режим связей, snap, ITU-T цвета, экспорт |
| `validation.ts` | Проверки перед сохранением |
| `exportDiagram.ts` | SVG/PNG схемы |

Данные сварки: `passport_data.splice_v1` на узле (`SpliceV1` в `muftaSpliceTypes.ts`) — связи `links[]`, метки волокон, занятость, waypoints.

### Режим связей

1. Включить «Режим связей» (`C` или чекбокс).
2. Клик по первому волокну, затем по второму (в т.ч. порт кросса `edgeId: 0`).
3. Опционально — промежуточные точки траектории связи (перетаскивание waypoint).

### Боковая панель

- Фильтр: все / занятые / свободные / со сваркой.
- Переход к волокну по номеру.
- Подпись владельца волокна, «в работе», цвет и стиль линии.
- Пакетная сварка диапазона волокон между двумя кабелями.
- Трассировка на схеме и на карте.

### Горячие клавиши (сварка)

| Клавиша | Действие |
|---------|----------|
| `C` | Режим связей |
| `F` | Сброс масштаба/панорамы |
| `Delete` / `Backspace` | Удалить выбранную связь |

### Сохранение

Валидация → `PUT /nodes/:id` с обновлённым `passport_data` и синхронизацией `used_fibers` на инцидентных участках.

## 8. База данных

Вкладка **База данных** — категории: проекты, ВОЛС, канализация, муфты, ТК, пикеты, кроссы.

- Поиск по названию, проекту, узлам.
- Сортировка (имя, id, длина, статус, дата).
- Фильтр по статусу ВОЛС (чипы).
- Клик — паспорт; двойной клик — на карте.
- Создание объектов (где доступно).

**Импорт/экспорт JSON** (настройки, только админ): **полная замена** всей базы (проекты, узлы, участки, заказы). Перед импортом сделайте «Скачать JSON» и сохраните копию.

**Импорт Excel** (настройки → «База данных (Excel)», только админ):

- **«Скачать шаблон Excel»** — файл `gis-import-template.xlsx` с листами `projects`, `nodes`, `edges`; на листе `nodes` несколько строк-примеров (массовое добавление).
- **Массовое добавление ТК/муфт:** строка 1 — заголовки колонок (`type`, `lat`, `lng`, …); строки 2, 3, 4… — по одному узлу в строке. Можно импортировать сразу много колодцев: достаточно **`type` + `lat` + `lng`**; `name` и поля паспорта можно оставить пустыми и заполнить в программе позже (пустое имя → «ТК-2», «Муфта-3» по номеру строки).
- **«Импорт из Excel…»** — режим **добавления** (`POST /database/import/append`): существующие объекты **не удаляются**; предпросмотр первых строк; при занятом `id` можно назначить новые автоматически (`remapTakenIds`). Проверяются **только новые строки** из файла.
- **Экспорт Excel** — «все ТК» или «ТК на карте» (видимая область).
- **GeoJSON** (Настройки → Интеграции) — импорт/экспорт колодцев `TK`.
- **Проверка качества базы** (админ) — `GET /database/health-report`.
- **Карта** — фильтры «только ТК», «сегодня»; автокластер при >200 узлах; viewport-загрузка при большой базе (`GET /map/viewport`); глобальный поиск через API (`/nodes?q=`, `/edges?q=`).
- В интерфейсе раскрыт блок **«Колонки и правила»** — для каждой колонки указано: обязательно при импорте, можно позже, только для MUFTA / ВОЛС.
- Парсер: [`frontend/src/import/excelImport.ts`](frontend/src/import/excelImport.ts).

| Способ | Действие |
|--------|----------|
| JSON | Полная замена базы |
| Excel | Только добавление новых строк |

**KML:** экспорт трасс выбранного проекта (`GET /export/kml`).

### Соответствие терминов и типов в JSON

| Как говорите в работе | Поле `type` в JSON | Раздел «База данных» |
|----------------------|--------------------|----------------------|
| Колодец, ТК, телефонная канализация (точка) | `TK` | ТК |
| Муфта | `MUFTA` | Муфты |
| Пикет | `PIKET` | Пикеты |
| Кросс | `KROSS` | Кроссы |
| Участок канализации (труба между колодцами) | `KANALIZACIYA` | Канализация |
| Участок ВОЛС (кабель) | `OPTOVOLOKNO` | ВОЛС |
| Проект | объект в `projects[]` | Проекты |

Координаты везде: **`lat`**, **`lng`** (широта, долгота). В `geometry` участка — массив точек **`[[lat, lng], [lat, lng], ...]`**.

### Общая обёртка файла

Любой импорт должен иметь такой корень (массивы могут быть пустыми `[]`):

```json
{
  "format": "gis-database",
  "version": 1,
  "projects": [],
  "nodes": [],
  "edges": []
}
```

**Как добавить только колодцы, не потеряв остальное:** Настройки → «Скачать JSON» → в массив `nodes` дописать новые объекты с **новыми уникальными `id`** → вставить **весь** файл в импорт. Пустые `projects` / `edges` в этом случае не подставляйте — скопируйте их из выгрузки как есть.

### Пример 1 — только колодцы (ТК), пустая база

Подходит, если вы начинаете с нуля и пока не нужны проекты, кабели и канализация.

```json
{
  "format": "gis-database",
  "version": 1,
  "projects": [],
  "nodes": [
    {
```

### Частые ошибки импорта

| Сообщение / ситуация | Причина |
|----------------------|---------|
| Пропали все кабели после импорта | В JSON не скопировали старые `edges` |
| `Дублируются id узлов` | Два узла с одним `id` |
| `Муфта … нужен parent_tk_id` | У `MUFTA` нет `parent_tk_id` |
| `слишком далеко от ТК` | Муфта дальше 2 м от своего ТК |
| `Канализация … только от ТК до ТК` | Канализация соединяет не ТК |
| `Оптика … нужен project_id` | Нет проекта или неверный id |
| `used_fibers не может быть больше total_fibers` | Ошибка в числах волокон |

Импорт: **Настройки** → раздел экспорта/импорта → вставить JSON или выбрать файл → импорт (роль **администратор**).

## 12.1 Совместная работа нескольких пользователей

Один инстанс backend (`http://host:4000`) обслуживает всех инженеров. Учётные записи хранятся в `system.sqlite`, сеть (проекты, узлы, участки) — в файлах `data/workspaces/<slug>.sqlite`.

### Рабочие базы (workspace)

| Кто | Действие |
|-----|----------|
| Любой авторизованный пользователь | `PUT /database/workspaces/mine` — выбрать **свою** базу (регион). Не влияет на коллег. |
| Администратор | `PUT /database/workspaces/active` — база **по умолчанию** для новых пользователей без личного выбора. |
| Администратор | Создание/удаление баз, импорт, резервные копии. |

На сервере открыт **пул подключений** SQLite (по `slug`), режим журнала **WAL**, `busy_timeout = 5000` мс — типичная нагрузка: 3–10 редакторов на одну базу.

### Синхронизация на карте

Клиент раз в ~12 с (настраивается в `userPrefs.workflow.syncPollIntervalSec`) запрашивает `GET /sync/summary`. При изменении `revision` выполняется обновление viewport и списков проектов/заказов. Если открыт паспорт объекта — показывается баннер «Обновить».

### Конфликты правок

`PUT /nodes/:id`, `PUT /edges/:id`, `PUT /projects/:id` принимают `expected_updated_at` из последнего чтения. При расхождении — **409** и сообщение «Объект изменён другим пользователем».

### Режим «только чтение»

Флаг `userDataReadOnly` в `app/config` блокирует мутации данных на API (кроме смены своего пароля).

### Журнал на сервере

`GET /activity?limit=100` (ADMIN) — события `create` / `update` / `delete` / `import` в `system.sqlite.activity_events`.

### Ограничения

- Не запускайте одновременно несколько тяжёлых полных импортов в одну базу.
- Восстановление из резервной копии затрагивает **вашу** открытую базу (slug из сессии админа) — предупредите коллег.
- При росте команды (>10 активных редакторов на одну базу) рассмотрите перенос на PostgreSQL (вне текущего MVP).

## 13. Рекомендации по разработке

### Общие принципы

1. **Минимальный diff** — менять только то, что нужно для задачи; не смешивать рефакторинг с фичей.
2. **Единый UI-слой** — новые кнопки через `gis-btn`, поля через `FormField` + `Input`/`Select`/`Textarea` (`frontend/src/ui/`). Стили — токены `--gis-*` в `index.css`, компонентные правила в `styles/ui.css` и `App.css`.
3. **Паспортные данные** — не писать «сырой» JSON в UI без схем: добавлять `read*` / `write*` в `passport/schemas.ts`.
4. **Сварка** — логика занятости волокон и связей в `muftaSpliceTypes.ts`; геометрия схемы — в `splice/layout.ts` и `spliceDiagram.tsx`; не дублировать расчёты в `App.tsx`.

### TypeScript и React

- Строгая типизация сущностей: `gisTypes.ts` (общие), локальные типы рядом с модулем (`splice/types.ts`, `users/types.ts`).
- Крупный state карты остаётся в `App.tsx`; вкладки получают данные и колбэки через props.
- `useMemo` / `useCallback` для тяжёлых списков (карта, layout сварки, аналитика).
- В JSX использовать стандартные HTML-теги (`div`, `span`); после правок прогонять `npm run build` — ловит несуществующие теги в разметке.

### Формы и вёрстка

- Подпись **над** полем: `FormField` или `.settings-field` / `.passport-fields` (колонка).
- Checkbox/radio — `.settings-choice` / `.passport-choice` (строка, не колонка).
- Не вешать глобальный `label { display: flex }` на целые вкладки — ломает настройки.

### Карта (Leaflet)

- Координаты в API: `[lat, lng]`; Leaflet: `[lat, lng]` как `LatLngExpression`.
- Стили линий ВОЛС — `opticalLinePathOptions` / статусы из `gisTypes.ts`.
- Новые слои — через `activeLayers` и фильтр `visibleEdges`, не отдельные копии массивов без необходимости.

### Backend

- Валидация входа — **Zod**-схемы в `server.js`; при добавлении полей обновлять схему и миграцию в `db.js`.
- Транзакции для операций, затрагивающих несколько таблиц (импорт, заказы, удаление проекта).
- Оптические концы: только `MUFTA` и `KROSS` (и алиасы на сервере); канализация — только `TK`.
- Муфта ↔ ТК: проверка расстояния `MUFTA_ATTACH_MAX_METERS` (2 м); несколько муфт на одном `parent_tk_id` — орбитальная раскладка (`backend/src/muftaTkLayout.js`).

### Сварка — типичные задачи

| Задача | Где править |
|--------|-------------|
| Расстояние между волокнами | `splice/layout.ts` (`MIN_FIBER_PITCH`, `fitPitchInSegment`) |
| Вид точки / hover | `spliceDiagram.tsx`, `spliceStyles.ts`, `.splice-fiber-vis--*` в CSS |
| Подписи #n и owner | `renderFiberPort` в `spliceDiagram.tsx`, координаты в `layout.ts` |
| Новая проверка перед сохранением | `splice/validation.ts` |
| Формат splice_v1 | `muftaSpliceTypes.ts` + миграция чтения старых данных |

### Тестирование перед PR

```bash
cd frontend && npm run build && npm run lint
cd backend && npm start   # smoke: health, login, CRUD
```

Ручной чеклист:

- [ ] Вход / выход, роли USER vs ADMIN
- [ ] Создание ТК → муфта → ВОЛС → паспорт → сварка → сохранение
- [ ] Импорт JSON (на копии БД)
- [ ] Тёмная тема и настройки карты
- [ ] Муфта с 24+ волокнами на схеме сварки

### Безопасность (production)

- Задать сильный `JWT_SECRET`.
- Не коммитить `backend/data/gis.sqlite` с реальными данными без необходимости.
- HTTPS перед выносом в сеть; CORS настроить под домен фронтенда.
- Импорт БД и восстановление бэкапа — только для доверенных администраторов.

### Git

- Коммиты — по запросу; не включать `node_modules`, секреты, случайные WAL/SHM SQLite без согласования.
